1. 💡 Como Usar o AI-DLC (Working with AI-DLC)

Este guia cobre como interagir efetivamente com o AI Agent em cada estágio — do primeiro prompt ao código funcionando. Baseado em experiências reais de workshops com equipes que adotaram a metodologia.

1.1 Regras Gerais

🔇 Explorar sem alterar documentos

Não envie perguntas exploratórias sem guardar com uma instrução de "não altere". O AI pode interpretar sua pergunta como uma mudança e atualizar documentos de design. Use o prefixo abaixo:

Do not update any documents. Help me understand why [this decision] was made.
Do not update any documents. For [component], is it reasonable to use [tech] here?
Do not change anything. Assess the impact of [proposed change].
💡 Dica: Comece toda mensagem exploratória com "Do not update any documents." Você pode remover essa restrição quando estiver pronto para agir.

📋 O fluxo Pergunta → Doc → Aprovação

O AI-DLC nunca faz perguntas inline no chat — ele escreve perguntas em arquivos markdown e aguarda você preencher as respostas. Isso mantém um registro durável de cada decisão.

1
AI cria o arquivo de perguntas

Ex.: aidlc-docs/inception/requirements/requirement-verification-questions.md — o AI para e espera você responder.

2
Você preenche as tags [Answer]

Use letra + label: [Answer]: B — ECS Fargate (containerized). Combine opções quando aplicável: [Answer]: B and C. Adicione ressalvas: [Answer]: B — migration is separate; include one-time data migration. Use X para resposta customizada.

3
Notifique o AI que as respostas estão prontas
We have answered your clarification questions. Please re-read the file and proceed.

Peça explicitamente para reler o arquivo — garante que o AI carregue suas respostas do disco.

4
AI valida e avança

O AI lê suas respostas, sinaliza ambiguidades restantes e prossegue para gerar o próximo artefato. Gates de aprovação sempre oferecem 2 opções: Request Changes ou Approve and Continue.

💡 Dica avançada: Se você tem documentação que já responde algumas perguntas: "Analyze the rationale for each question. If a question has already been answered through the provided documentation, answer it yourself. Only ask me if it is still unclear."

🧹 Gerenciamento de Contexto

O AI-DLC depende de ter a cadeia completa de artefatos em contexto para gerar saídas consistentes. Contexto acumulado degrada a qualidade de forma sutil e difícil de diagnosticar.

A regra central: limpe o contexto em cada ponto de decisão natural (gate).

  • Quando o AI pede para responder um arquivo de perguntas → responda, depois inicie contexto fresco
  • Quando o AI apresenta um documento para aprovação → revise, depois inicie contexto fresco
  • Se sua ferramenta oferece "compact context" no meio do workflow → sempre recuse

Como retomar após reset de contexto:

Go to aidlc-docs/aidlc-state.md, find the first unchecked item, then go to the corresponding plan file and resume from that point.
💡 Dica: Faça commit e push de todas as mudanças antes de cada reset de contexto. Leva segundos e garante sempre um ponto de recuperação limpo.

📦 Batching de Prompts

Quando duas mudanças são fortemente acopladas ao mesmo assunto: inclua as duas num único prompt.
Quando duas mudanças são não-relacionadas: faça uma de cada vez.

Over-batching (combinar mudanças não-relacionadas) faz o AI perder foco e deixar detalhes passar. Under-batching (prompts separados para coisas intimamente relacionadas) adiciona round-trips desnecessários.

🔍 Critiques Independentes

O AI defenderá suas próprias decisões anteriores. Para avaliação imparcial de um artefato, peça uma crítica em um contexto fresco — onde o AI não tem memória de por que tomou aquelas decisões.

Produce a critique document of [the requirements document / the component design]. Do this in a new context separate from everything else.

1.2 Fase INCEPTION — Interações Específicas

🚀 Iniciando um projeto

Com seus documentos de entrada prontos (Vision + Technical Environment):

I want to start a new project. Please read [path to vision document] and [path to technical environment document], then begin the AIDLC workflow.

O AI escaneará o workspace, determinará greenfield vs. brownfield, e avançará para Requirements Analysis usando seus documentos como fonte primária.

📝 Respondendo perguntas de Requirements Analysis

  • Separe Full Vision do MVP explicitamente. Se o AI perguntar quais features incluir, nomeie-as. Se algo está fora do escopo, diga claramente.
  • Declare decisões de "não" deliberadas. D — no caching required at this time sinaliza intenção. Uma resposta vazia convida o AI a fazer uma escolha especulativa.
  • Descreva abordagens em fases inline. X — simple role-based workflow now; replace with external workflow engine when available
💡 Extensões de Segurança: Durante Requirements Analysis, o AI perguntará se você quer aplicar regras de extensão de segurança. Para aplicações de produção, escolha Sim. Para protótipos, Não é adequado. Essa decisão é registrada e aplicada em toda a Construction Phase.

🔧 Outras interações úteis na Inception

Mover feature para backlog mid-stream:

We are going to backlog the [feature name] capability for the current release. Please remove it from the component design and flag the related user stories as backlogged.

Verificar impacto upstream após mudança de design:

Now review the previous steps — user stories and requirements — to ensure this change does not require updates to any of those documents.
💡 Regra standing de back-propagation: Configure no início da fase: "Every time you update a document, check whether the change impacts the requirements document and user stories, and prompt me if it does." Isso cria uma rede de segurança automática.

Review paralelo por times:

Restrict your edits to the files under your team's control. When all teams are done, we will ask the AI to review all changes and confirm there are no conflicts. Then we will ask it to review impacts to user stories and requirements.
We had [N] independent groups editing component design files. Please review all files and report any conflicts or inconsistencies. Do not edit the files — produce a report.

1.3 Fase CONSTRUCTION — Interações Específicas

🏗️ Transição Design → Code Generation

We have completed component design review. We are ready for code creation. Please use the following directory and source code structure: [reference an existing service or folder structure]. Use this pattern for APIs. For the UI, follow the [Vue.js composables/components/store] directory structure. Please ask any questions you have before proceeding.

Convidar perguntas antes de gerar resolve ambiguidades no plano em vez de no meio da criação de arquivos.

💻 Code Generation — Boas Práticas

Parte 1 — Planning: Antes de aprovar o plano, verifique:

  • Todos os arquivos estão no lugar certo (código de aplicação na raiz do workspace, nunca em aidlc-docs/)
  • Os passos cobrem tudo que os documentos de design especificaram
  • Projetos brownfield listam arquivos existentes a modificar (não cópias novas ao lado)
💡 Bibliotecas internas: Antes de aprovar o plano, injete seus requisitos de biblioteca interna: "In addition to my answers, you must use the following libraries from our starter project: [list each]. Explain why and when each should be used."

Correção direcionada:

The [endpoint] should use [correct parameter], not [incorrect parameter]. Please update the [component name] accordingly.

Brownfield — arquivo duplicado indesejado:

I see [ClassName_modified.java] alongside [ClassName.java]. Please merge the changes into the original file and delete the duplicate.

🔄 Back-propagação após Code Generation

Mudanças feitas durante a geração de código precisam ser propagadas de volta aos documentos de design. Faça isso como uma varredura deliberada após o polimento do código:

When you have finished polishing the code, review each unit's final design files and propagate any changes back up the chain to requirements and user stories. Make a plan for how to do this step by step before executing.

🧪 Build and Test — Dicas

Injetar tooling de teste no momento certo (não no início do projeto — pode se perder nas etapas intermediárias):

At the functional test generation step, inject the following instruction: generate functional tests using the [test management system] format described in this document: [attach specification].

Escopo de cobertura de testes unitários:

When generating unit tests, exclude third-party external dependencies from code coverage calculations. Require a minimum of 80% coverage on internal code paths only.

1.4 ⚠️ Nunca "Vibe Code"

⛔ Regra fundamental: Nunca edite diretamente arquivos de código gerado para correções rápidas — isso cria drift entre o código e os documentos de design. O AI-DLC usará os documentos de design como fonte de verdade em toda operação subsequente. Se o código e os documentos divergem, a próxima geração causa confusão e retrabalho.

O jeito correto de fazer uma mudança

1
Descreva o problema sem tocar em nada
Do not update any documents yet. I have discovered issue [X]. Review the design and help me understand where this needs to be addressed.
2
Corrija o documento de design primeiro
Please update [specific design document] to reflect [the fix]. Then check whether any upstream documents — requirements, user stories — also need updating.
3
Regenere apenas o código afetado
The design for [unit name] has been updated. Please re-run code generation for the affected files only.

Se você teve que editar diretamente (emergência):

We made a temporary direct edit to [file] to unblock the team. The fix was [description]. Please update [design document] to reflect this and verify no other documents are inconsistent.

🛡️ Regras standing que previnem drift (configure no início da Construction Phase)

Every time you update a document, check whether the change impacts the requirements document and user stories, and prompt me if it does.
When you make a design decision during code generation, always make sure the documentation reflects this change before proceeding.

📁 Mantenha relatórios fora do aidlc-docs/

Relatórios voltados a humanos (diagramas de arquitetura, sumários, apresentações) não devem ser salvos em aidlc-docs/ — serão carregados como artefatos em estágios subsequentes, inflando o token count e confundindo o AI sobre o que é input de design autoritativo. Use uma pasta separada:

Pause the process. Start a new context. Read [report specification file] and produce the report based on the current state of the AIDLC artifacts. Save the output to a reports/ folder, not aidlc-docs/.

2. Visão Geral do AI-DLC

O AI-DLC (AI-Driven Development Life Cycle) é uma metodologia de desenvolvimento de software guiada por IA que se adapta à complexidade do projeto, mantém padrões de qualidade e garante que o ser humano permaneça no controle do processo. É dividida em três fases principais:

FaseFocoObjetivo
🔵 INCEPTIONO QUE e POR QUÊPlanejamento, levantamento de requisitos e decisões arquiteturais
🟢 CONSTRUCTIONCOMO construirDesign detalhado, implementação NFR e geração de código
🟡 OPERATIONSCOMO operarImplantação e monitoramento (expansão futura / placeholder)
🧠

Execução Adaptativa

Apenas os estágios que agregam valor ao projeto específico são executados — o workflow se adapta ao trabalho.

👁️

Planejamento Transparente

Um plano de execução é sempre apresentado e aprovado pelo usuário antes do início de qualquer estágio.

🙋

Human in the Loop

O agente propõe, o humano aprova. Aprovação explícita é exigida em cada estágio crítico — sem autoproceder.

📝

Trilha de Auditoria

Todos os inputs do usuário e outputs do AI são registrados em audit.md com timestamps ISO 8601.

🔌

Agnóstico de Plataforma

Funciona com qualquer IDE ou agente: Claude Code, Amazon Q, Cursor, Kiro, GitHub Copilot, OpenAI Codex.

🧩

Extensível

Sistema de extensões permite adicionar regras de segurança, compliance e padrões organizacionais sobre o core.

3. 📥 Inputs Pré-INCEPTION — Documentos de Entrada

Antes de iniciar o workflow AI-DLC, a metodologia recomenda fortemente a preparação de dois documentos de entrada. Eles reduzem drasticamente o número de perguntas de clarificação na Requirements Analysis e garantem que o AI Agent parta do contexto real da equipe — não de suposições.

💡 Dica principal: Cada lacuna preenchida nesses documentos é uma pergunta a menos durante a Requirements Analysis. Quanto mais contexto você fornecer, mais preciso e eficiente será o workflow.
📄 Vision Document Guide

Define os objetivos de negócio, resultados esperados e limites de escopo antes de entrar no AI-DLC. É a principal entrada para a Inception Phase.

Quando escrever
  • Antes de qualquer projeto novo ou iniciativa maior
  • Ao propor um novo produto, conjunto de features ou plataforma
  • Quando há pivô de direção de produto existente
  • Quando múltiplos stakeholders precisam se alinhar antes do desenvolvimento
Estrutura do documento
  • 1. Executive Summary — 3–5 frases: o que é, para quem, por quê existe
  • 2. Business Context — Problem Statement, Business Drivers, Target Users, Constraints, Success Metrics
  • 3. Full Scope Vision — visão aspiracional completa do produto; feature areas, integrações, user journeys, roadmap
  • 4. MVP Scope — menor conjunto que entrega valor: features IN scope, features OUT of scope explícitas, critérios de sucesso testáveis, Definition of Done
  • 5. Risks and Dependencies — riscos com likelihood/impact/mitigation, dependências externas, open questions
Mínimo viável para começar rápido
  • 1 parágrafo: o que você está construindo e para quem
  • Lista de features do MVP em escopo
  • Lista de features explicitamente FORA do escopo
  • Open questions (ambiguidades já conhecidas)
📌 As open questions alimentam diretamente a Requirements Analysis como ambiguidades pré-declaradas, resolvidas antes de surpreender o design.
Diretrizes de escrita
  • ✅ Seja específico e mensurável ("reduzir tempo de processamento em 30%")
  • ✅ Separe claramente Full Vision do MVP
  • ✅ Inclua listas de "fora do escopo" — são tão valiosas quanto as listas de escopo
  • ❌ Não use linguagem vaga: "world-class", "seamless", "intuitive"
  • ❌ Não inclua detalhes de tecnologia/implementação (isso vai no Technical Environment)
  • ❌ Não pule a seção de MVP
⚙️ Technical Environment Document Guide

Define tooling técnico, padrões, restrições e preferências que governam como o projeto é construído. É a referência vinculante durante a Construction Phase.

Aplica-se a dois contextos: Greenfield (do zero — todas as escolhas estão abertas) e Brownfield (codebase existente — escolhas são restritas pelo que existe).

Estrutura do documento
  • 1. Project Technical Summary — nome, tipo, cloud provider, deployment model, tamanho e experiência da equipe
  • 2. Programming Languages — Required / Permitted / Prohibited (com razão para cada proibição)
  • 3. Frameworks and Libraries — Required / Preferred / Prohibited (com alternativa preferida), processo de aprovação de nova lib
  • 4. Cloud Environment — provider, regiões, Service Allow List (com constraints), Service Disallow List (com alternativa), processo de aprovação
  • 5. Preferred Technologies and Patterns — padrões de arquitetura, API design standards, data patterns, messaging, frontend
  • 6. Security Requirements — auth/authz, data protection, network security, secrets management, compliance framework (OWASP, NIST, etc.)
  • 7. Testing Requirements — tipos de teste, coverage mínimo, tooling por tipo, CI/CD gates
  • 8. Example & Template Code — código de exemplo por padrão (endpoint, repositório, teste, IaC) que o AI usará como referência durante Code Generation
  • 9. Brownfield-Specific (se aplicável) — inventário atual, regras de migração (Keep/Migrate/Remove), regras de coexistência
Mínimo viável para começar rápido
  • Linguagem e versão, gerenciador de pacotes
  • Web framework, cloud provider, deployment model, test framework
  • Tabela de bibliotecas proibidas (com razão e alternativa recomendada)
  • Segurança básica (auth method, secrets policy)
  • Ao menos 1 exemplo de: endpoint típico, função e teste
🏆 Maior valor: Os exemplos de código são a adição de maior impacto além do básico — dão ao AI um padrão concreto a seguir durante a Code Generation em vez de inventar o próprio.
⚠️ A tabela de bibliotecas proibidas importa mais que uma lista simples. As colunas de razão e alternativa ensinam ao AI-DLC por que uma lib é proibida, levando a melhores decisões de substituição.

4. 🔵 INCEPTION PHASE

Determina O QUE construir e POR QUÊ. Abrange desde a detecção do workspace até a decomposição do sistema em unidades de trabalho mensuráveis.

2.1 Workspace Detection SEMPRE

🔍 Workspace Detection — Análise do Estado do Workspace
🎯 Propósito
Determinar o estado do workspace: verificar se já existe um projeto AI-DLC em andamento (retomada de sessão) ou iniciar novo. Classifica o projeto como Greenfield (sem código existente) ou Brownfield (com código existente), o que determina o próximo estágio.
🔀 Condição de Execução
SEMPRE executado — é o primeiro estágio de qualquer workflow AI-DLC. Não tem pré-requisitos.
⛔ Gate de Saída
Automático, sem aprovação — este estágio é apenas informativo. Avança automaticamente para Reverse Engineering (brownfield sem artefatos) ou Requirements Analysis (greenfield ou com artefatos existentes).
📄 Artefatos Gerados
• aidlc-docs/aidlc-state.md — criado com tipo de projeto, estado do workspace, caminho raiz e regra de localização de código
• Entrada inicial no audit.md com o input completo do usuário e timestamp
👥 Papéis Envolvidos
• AI Agent (papel principal — executa varredura e análise automática)
• Developer / Product Owner (fornece contexto inicial via requisição)

2.2 Reverse Engineering CONDICIONAL

🔬 Reverse Engineering — Análise do Codebase Existente
🎯 Propósito
Analisar a base de código existente (brownfield) e gerar artefatos abrangentes de documentação: visão de negócio, arquitetura, estrutura de código, APIs, tecnologias e qualidade. Esses artefatos servem de contexto para todos os estágios subsequentes.
🔀 Condição de Execução
Executa se: projeto Brownfield detectado E sem artefatos anteriores de RE (ou artefatos desatualizados).
Skip se: projeto Greenfield, ou artefatos de RE existentes e ainda válidos.
⛔ Gate de Saída
Aprovação explícita obrigatória — o usuário revisa os artefatos gerados e pode solicitar alterações ou aprovar e avançar para Requirements Analysis. Não há avanço automático.
📄 Artefatos Gerados
• business-overview.md — visão de negócio, transações e dicionário
• architecture.md — diagrama de arquitetura e descrição de componentes
• code-structure.md — estrutura, padrões de design, inventário de arquivos
• api-documentation.md — endpoints REST e APIs internas com modelos de dados
• component-inventory.md — inventário completo de pacotes/componentes
• technology-stack.md — linguagens, frameworks e ferramentas
• dependencies.md — dependências internas e externas com diagrama Mermaid
• code-quality-assessment.md — cobertura de testes, dívida técnica, padrões
• reverse-engineering-timestamp.md — metadados e checklist de artefatos gerados
👥 Papéis Envolvidos
• AI Agent (papel principal — analisa codebase e gera documentação)
• Software Architect (revisa artefatos e valida precisão da arquitetura)
• Tech Lead / Senior Developer (valida precisão técnica da documentação)

2.3 Requirements Analysis SEMPRE

📋 Requirements Analysis — Levantamento Adaptativo de Requisitos
🎯 Propósito
Levantar, analisar e documentar requisitos funcionais e não-funcionais. Identifica intenção do usuário, tipo de requisição (nova feature, bug fix, refactoring, etc.), escopo estimado e complexidade. Garante que todas as ambiguidades sejam eliminadas antes do planejamento. Também apresenta perguntas de opt-in para extensões (segurança, compliance).
🔀 Condição de Execução
SEMPRE executado, com profundidade adaptativa:
• Minimal: requisição simples e clara — apenas documenta intenção
• Standard: complexidade normal — requisitos funcionais e não-funcionais
• Comprehensive: alta complexidade/risco — requisitos detalhados com rastreabilidade
⛔ Gate de Saída
Gate duplo obrigatório:
1. Usuário deve responder todas as perguntas em requirement-verification-questions.md (tags [Answer])
2. Nenhum avanço até ambiguidades resolvidas via follow-up questions
3. Aprovação final do documento de requisitos antes de avançar
📄 Artefatos Gerados
• requirement-verification-questions.md — perguntas com tags [Answer] respondidas pelo usuário
• requirements.md — documento completo de requisitos funcionais e não-funcionais, com análise de intenção, tipo, escopo e complexidade
• Configuração de extensões em aidlc-state.md (tabela Extension Configuration)
• Atualização do aidlc-state.md e registro no audit.md
👥 Papéis Envolvidos
• AI Agent (assume papel de Product Owner — formula e valida requisitos)
• Business Stakeholders (respondem às perguntas de clarificação)
• Developer / Tech Lead (fornecem contexto técnico quando necessário)
• End Users (indiretamente, através das histórias e personas)

2.4 User Stories CONDICIONAL

📚 User Stories — Narrativas Centradas no Usuário
🎯 Propósito
Traduzir requisitos de negócio em narrativas centradas no usuário, com personas bem definidas e critérios de aceite claros (critério INVEST). Aumenta o alinhamento entre equipes, fornece especificações testáveis e reduz riscos de implementação incorreta.
🔀 Condição de Execução
Alta prioridade (sempre executa): novas funcionalidades para o usuário, mudanças em fluxos existentes, múltiplas personas, APIs customer-facing, lógica de negócio complexa, projetos cross-team.

Skip apenas para: refatoração pura sem impacto ao usuário, bug fixes simples e isolados, mudanças de infraestrutura sem efeito no front-end, melhorias em ferramentas de desenvolvedor. Em caso de dúvida, inclua o estágio.
⛔ Gate de Saída
Duas aprovações separadas obrigatórias:
• Parte 1 (Planning): usuário responde perguntas no plano; ambiguidades resolvidas antes de gerar
• Parte 2 (Generation): artefatos gerados (stories + personas) aguardam aprovação final
📄 Artefatos Gerados
• user-stories-assessment.md — justificativa documentada para execução do estágio
• story-generation-plan.md — plano com abordagem escolhida e perguntas [Answer]
• stories.md — histórias de usuário no formato INVEST com critérios de aceite por história
• personas.md — arquétipos de usuário com características e motivações
• Atualização do aidlc-state.md e registro no audit.md
👥 Papéis Envolvidos
• AI Agent (assume papel de Product Owner — cria histórias e personas)
• Business Stakeholders / Patrocinadores (validam histórias e critérios de aceite)
• UX Designer (contribui com personas e fluxos de usuário)
• QA Engineer (revisa testabilidade dos critérios de aceite)
• Developers (estimam complexidade e negociam escopo das histórias)

2.5 Workflow Planning SEMPRE

📊 Workflow Planning — Plano de Execução Completo
🎯 Propósito
Determinar quais fases executar e criar um plano de execução abrangente. Realiza análise detalhada de impacto de mudança, avaliação de risco e mapeamento de dependências entre componentes. Gera visualização Mermaid do fluxo completo e documenta justificativas para cada decisão de EXECUTE/SKIP.
🔀 Condição de Execução
SEMPRE executado — ocorre após Requirements Analysis (e User Stories, se aplicável). Nenhum pré-requisito adicional.
⛔ Gate de Saída
Aprovação obrigatória do plano — usuário revisa o plano de execução, pode adicionar estágios marcados como SKIP ou solicitar modificações. Aprovação explícita antes de avançar para o próximo estágio definido no plano.
📄 Artefatos Gerados
• execution-plan.md contendo:
  - Análise de impacto de mudança (user-facing, structural, data model, API, NFR)
  - Avaliação de risco: Low / Medium / High / Critical
  - Mapa de dependências entre componentes (brownfield)
  - Estratégia de atualização de módulos (sequential/parallel/hybrid)
  - Visualização Mermaid do fluxo com estágios EXECUTE/SKIP coloridos
  - Lista completa de estágios com rationale para cada decisão
• aidlc-state.md com plano completo e critérios de sucesso
• Registro no audit.md
👥 Papéis Envolvidos
• AI Agent (papel principal — analisa contexto e determina plano ótimo)
• Solution Architect (revisa decisões de escopo e impacto técnico)
• Project Manager (valida cronograma e prioridades do projeto)
• Product Owner (aprova o plano de execução e alinha com visão de produto)

2.6 Application Design CONDICIONAL

🏗️ Application Design — Design de Alto Nível de Componentes
🎯 Propósito
Identificar e definir os principais componentes funcionais, suas responsabilidades, interfaces de alto nível e dependências. Projeta a camada de serviço para orquestração. Nota: lógica de negócio detalhada é definida no Functional Design da CONSTRUCTION PHASE.
🔀 Condição de Execução
Executa se: novos componentes/serviços são necessários; métodos e regras precisam ser definidos; design de service layer é requerido; dependências precisam ser esclarecidas.
Skip se: mudanças dentro de componentes existentes sem novos componentes ou métodos.
⛔ Gate de Saída
Gate duplo: usuário responde perguntas no plano (design de componentes, métodos, serviços e dependências); todas ambiguidades resolvidas; aprovação final dos artefatos gerados.
📄 Artefatos Gerados
• application-design-plan.md — plano com perguntas [Answer] respondidas
• components.md — definição de componentes e responsabilidades
• component-methods.md — assinaturas de métodos e interfaces (alto nível)
• services.md — definições de serviços e padrões de orquestração
• component-dependency.md — matriz de dependências e padrões de comunicação
• application-design.md — documento consolidado de todos os artefatos acima
• Atualização do aidlc-state.md e registro no audit.md
👥 Papéis Envolvidos
• AI Agent (assume papel de Software Architect — define estrutura de componentes)
• Tech Lead / Senior Developer (valida decisões de design e limites de componentes)
• Product Owner (alinha design com requisitos de negócio)
• Domain Experts (contribuem com conhecimento de domínio de negócio)

2.7 Units Generation CONDICIONAL

🔧 Units Generation — Decomposição em Unidades de Trabalho
🎯 Propósito
Decompor o sistema em unidades de trabalho gerenciáveis com limites claros e rastreabilidade de histórias. Para microsserviços: cada unidade torna-se um serviço implantável independentemente. Para monolitos: unidade única com módulos lógicos. Define a sequência de desenvolvimento da CONSTRUCTION PHASE.
🔀 Condição de Execução
Executa se: decomposição em múltiplas unidades é necessária; múltiplos serviços/módulos; sistema complexo. Pré-requisito: Application Design concluído.
Skip se: unidade única simples; nenhuma decomposição necessária.
⛔ Gate de Saída
Duas aprovações separadas:
• Parte 1 (Planning): perguntas de decomposição respondidas e ambiguidades resolvidas
• Parte 2 (Generation): artefatos de unidades aprovados antes de avançar à CONSTRUCTION
📄 Artefatos Gerados
• unit-of-work-plan.md — plano de decomposição com perguntas [Answer]
• unit-of-work.md — definição de cada unidade com responsabilidades e estratégia de organização de código
• unit-of-work-dependency.md — matriz de dependências entre unidades
• unit-of-work-story-map.md — mapeamento de histórias de usuário para unidades
• Atualização do aidlc-state.md e registro no audit.md
👥 Papéis Envolvidos
• AI Agent (assume papel de Solution Architect — decompõe o sistema)
• Tech Lead (valida limites de unidade e dependências técnicas)
• Team Leads (contribuem com alinhamento de equipe e ownership de serviços)
• DevOps Engineer (contribui com requisitos de implantação por unidade)

5. 🟢 CONSTRUCTION PHASE

Determina COMO construir. Executa um loop completo de design + código para cada unidade de trabalho. Após todas as unidades, executa Build and Test unificado.

🔄 Loop por Unidade de Trabalho

  • Para cada unidade: Functional Design → NFR Requirements → NFR Design → Infrastructure Design → Code Generation
  • Cada unidade é completada integralmente antes de avançar para a próxima
  • Após TODAS as unidades: Build and Test (executado uma única vez ao final)

3.1 Functional Design CONDICIONAL por unidade

🔧 Functional Design — Design de Lógica de Negócio por Unidade
🎯 Propósito
Design detalhado da lógica de negócio por unidade, tecnologicamente agnóstico. Modela entidades de domínio, regras de negócio, algoritmos, fluxos de dados e cenários de erro. Baseia-se no Application Design da INCEPTION. Quando inclui frontend, detalha hierarquia de componentes UI, props/state e interações.
🔀 Condição de Execução
Executa se: novos modelos de dados; lógica de negócio complexa; regras de negócio precisam de design detalhado. Pré-requisito: Units Generation concluído.
Skip se: mudanças de lógica simples; nenhuma nova lógica de negócio.
⛔ Gate de Saída
Gate obrigatório com 2 opções fixas: "Solicitar Alterações" ou "Continuar para o Próximo Estágio". Nenhuma terceira opção é permitida — prevenção de comportamento emergente do AI.
📄 Artefatos Gerados
• {unit-name}-functional-design-plan.md — plano com perguntas [Answer]
• business-logic-model.md — modelo de lógica de negócio da unidade
• business-rules.md — regras de negócio, validações e restrições detalhadas
• domain-entities.md — entidades de domínio, relacionamentos e atributos
• frontend-components.md (se aplicável) — hierarquia UI, props/state, fluxos de interação
👥 Papéis Envolvidos
• AI Agent (Software Engineer / Domain Expert — projeta lógica de negócio)
• Tech Lead (valida lógica e aderência ao design de alto nível)
• Business Analyst (confirma regras de negócio com stakeholders)
• QA Engineer (revisa cenários de teste e edge cases)
• UX Designer (quando frontend components aplicável)

3.2 NFR Requirements CONDICIONAL por unidade

📊 NFR Requirements — Requisitos Não-Funcionais por Unidade
🎯 Propósito
Determinar os requisitos não-funcionais da unidade e definir as decisões de tech stack. Abrange escalabilidade, performance, disponibilidade, segurança, confiabilidade e manutenibilidade. As decisões de NFR guiam diretamente o NFR Design.
🔀 Condição de Execução
Executa se: requisitos de performance existem; considerações de segurança necessárias; preocupações de escalabilidade; seleção de tech stack necessária. Pré-requisito: Functional Design concluído.
Skip se: setup NFR existente é suficiente; nenhum novo NFR; mudanças simples.
⛔ Gate de Saída
Gate obrigatório com 2 opções fixas: "Solicitar Alterações" ou "Continuar para o Próximo Estágio". Aprovação explícita exigida.
📄 Artefatos Gerados
• {unit-name}-nfr-requirements-plan.md — plano com perguntas [Answer]
• nfr-requirements.md — requisitos de escalabilidade, performance, disponibilidade, segurança, confiabilidade, manutenibilidade e usabilidade
• tech-stack-decisions.md — decisões de stack tecnológica com justificativas detalhadas
👥 Papéis Envolvidos
• AI Agent (Solutions Architect — define e analisa NFRs)
• DevOps / Platform Engineer (contribui com requisitos operacionais e SLAs)
• Security Engineer (define requisitos de segurança e compliance)
• Performance Engineer (define SLAs e benchmarks de performance)
• Product Owner (alinha expectativas de negócio com capacidade técnica)

3.3 NFR Design CONDICIONAL por unidade

🎨 NFR Design — Padrões de Design para Requisitos Não-Funcionais
🎯 Propósito
Incorporar os requisitos não-funcionais ao design da unidade através de padrões arquiteturais e componentes lógicos. Define padrões de resiliência (retry, circuit breaker, fallback), escalabilidade (sharding, partitioning), performance (caching, async) e segurança. Os componentes lógicos definidos aqui guiam o Infrastructure Design.
🔀 Condição de Execução
Executa se: NFR Requirements foi executado E padrões NFR precisam ser incorporados ao design.
Skip se: nenhum NFR Requirements; NFR Requirements foi ignorado.
⛔ Gate de Saída
Gate obrigatório com 2 opções fixas: "Solicitar Alterações" ou "Continuar para o Próximo Estágio". Aprovação explícita exigida.
📄 Artefatos Gerados
• {unit-name}-nfr-design-plan.md — plano com perguntas [Answer]
• nfr-design-patterns.md — padrões de design por categoria: resiliência, escalabilidade, performance, segurança
• logical-components.md — componentes lógicos de infraestrutura (filas, caches, circuit breakers, load balancers) e padrões de integração
👥 Papéis Envolvidos
• AI Agent (Solutions Architect — aplica padrões NFR ao design)
• Software Architect (valida padrões de design aplicados e sua adequação)
• DevOps / SRE Engineer (revisa componentes de resiliência e observabilidade)
• Security Architect (revisa padrões e controles de segurança)

3.4 Infrastructure Design CONDICIONAL por unidade

🏢 Infrastructure Design — Mapeamento para Infraestrutura Real
🎯 Propósito
Mapear componentes de software lógicos (definidos no NFR Design) para serviços reais de infraestrutura em ambientes de implantação (AWS, Azure, GCP, on-premise). Define serviços de compute, storage, messaging, networking, monitoramento e alertas para cada unidade.
🔀 Condição de Execução
Executa se: serviços de infraestrutura precisam de mapeamento; arquitetura de implantação necessária; recursos de cloud precisam ser especificados. Pré-requisito: Functional Design (NFR Design recomendado).
Skip se: nenhuma mudança de infraestrutura; infraestrutura já definida.
⛔ Gate de Saída
Gate obrigatório com 2 opções fixas: "Solicitar Alterações" ou "Continuar para Code Generation". Aprovação explícita exigida.
📄 Artefatos Gerados
• {unit-name}-infrastructure-design-plan.md — plano com perguntas [Answer]
• infrastructure-design.md — mapeamento de componentes lógicos para serviços reais de cloud (compute, storage, messaging, networking, monitoramento)
• deployment-architecture.md — arquitetura de implantação com decisões e justificativas
• shared-infrastructure.md (se aplicável) — recursos de infraestrutura compartilhados entre unidades
👥 Papéis Envolvidos
• AI Agent (Cloud/Infrastructure Architect — mapeia para serviços de cloud)
• DevOps / Platform Engineer (especifica e valida recursos de infraestrutura)
• Security Engineer (valida configurações de rede, IAM e segurança)
• FinOps Engineer (avalia custo dos recursos de cloud propostos)
• Solutions Architect (alinha com arquitetura de referência da organização)

3.5 Code Generation SEMPRE por unidade

💻 Code Generation — Geração de Código, Testes e Artefatos
🎯 Propósito
Gerar código completo, testes unitários e artefatos de implantação para cada unidade, seguindo um plano explícito com rastreabilidade total para histórias de usuário. Em projetos brownfield, modifica arquivos existentes in-place (nunca cria cópias com sufixos _modified). Garante elementos amigáveis a automação de testes (data-testid em UI).
🔀 Condição de Execução
SEMPRE executado para cada unidade — é o estágio central da CONSTRUCTION PHASE. Requer que os estágios de design da unidade estejam concluídos.
⛔ Gate de Saída
Duas aprovações separadas:
• Parte 1 (Planning): plano detalhado com checkboxes e rastreabilidade de histórias aprovado antes de gerar
• Parte 2 (Generation): código gerado revisado e aprovado antes de avançar para próxima unidade ou Build & Test
📄 Artefatos Gerados
• {unit-name}-code-generation-plan.md — plano detalhado com checkboxes por step
• Código da aplicação (na raiz do workspace — NUNCA em aidlc-docs/):
  - Camada de lógica de negócio + testes unitários
  - Camada API (REST/GraphQL/gRPC) + testes unitários
  - Camada de repositório/persistência + testes unitários
  - Componentes frontend + testes (se aplicável, com data-testid)
  - Scripts de migração de banco de dados
  - Artefatos de implantação (Dockerfile, IaC, scripts)
  - Documentação técnica (README, API docs)
• Documentação em aidlc-docs/construction/{unit-name}/code/ (apenas resumos markdown)
• aidlc-state.md com histórias marcadas como concluídas [x]
👥 Papéis Envolvidos
• AI Agent (Software Engineer — gera código seguindo o plano)
• Tech Lead (revisa qualidade, padrões e aderência ao design)
• QA Engineer (verifica cobertura e qualidade dos testes)
• Security Engineer (revisa segurança do código gerado)
• DevOps Engineer (valida artefatos de implantação e scripts IaC)

3.6 Build and Test SEMPRE — após todas as unidades

🔨 Build and Test — Estratégia Abrangente de Build e Testes
🎯 Propósito
Gerar instruções completas de build e execução de testes para todas as unidades combinadas. Define estratégia de testes multicamada: unitários, integração, performance, contratos (microsserviços), segurança e end-to-end. Prepara o projeto para a fase de Operations.
🔀 Condição de Execução
SEMPRE executado — ocorre após Code Generation de todas as unidades estar concluído.
⛔ Gate de Saída
Aprovação obrigatória antes de avançar para Operations. Usuário revisa o sumário de build e test e confirma prontidão.
📄 Artefatos Gerados
• build-instructions.md — pré-requisitos, passos de build, troubleshooting
• unit-test-instructions.md — execução de testes unitários e cobertura esperada
• integration-test-instructions.md — cenários de integração entre unidades/serviços
• performance-test-instructions.md (se aplicável) — testes de carga/stress, SLAs
• contract-test-instructions.md (para microsserviços) — validação de contratos de API
• security-test-instructions.md — vulnerability scanning, testes de autenticação/autorização
• e2e-test-instructions.md — fluxos completos de usuário
• build-and-test-summary.md — sumário consolidado com status de build e todos os testes
👥 Papéis Envolvidos
• QA Engineer (papel central — define e executa estratégia de testes)
• DevOps / Build Engineer (gerencia pipeline de build e CI/CD)
• Performance Engineer (conduz testes de carga e stress)
• Security Engineer (conduz testes e scanning de segurança)
• Tech Lead (revisa resultados e aprova prontidão para produção)
• Product Owner (valida que critérios de aceite foram atendidos)

6. 🟡 OPERATIONS PHASE

Determina COMO implantar e operar o sistema. Atualmente é um placeholder para expansão futura. Todas as atividades de build e teste são tratadas na CONSTRUCTION PHASE.

4.1 Operations PLACEHOLDER

🚀 Operations — Implantação e Operações (Expansão Futura)
🎯 Propósito
Gerenciar implantação em produção, monitoramento contínuo e operações do sistema. Esta fase representa o ciclo DevOps completo, fechando o loop de feedback para o desenvolvimento. Atualmente reservada para expansão futura da metodologia.
🔀 Condição de Execução
PLACEHOLDER — nenhum estágio ativo atualmente. Status atual: reservado para expansão futura.
⛔ Gate de Saída
A ser definido nas versões futuras da metodologia.

• Checklists de prontidão para produção
• Workflows de manutenção, suporte e on-call
• Estratégias de rollback e disaster recovery
👥 Papéis Planejados

• Product Owner (validação final de prontidão para produção)
